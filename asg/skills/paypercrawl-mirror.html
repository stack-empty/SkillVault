# CrawlGuard Security Documentation

## Overview

CrawlGuard implements defense-in-depth security across the WordPress plugin and Cloudflare Worker backend. This document describes each security layer.

---

## 1. Authentication

### API Key Authentication
- All authenticated endpoints require `X-API-Key` header
- Keys are prefixed `cg_live_` (40 hex chars) for easy identification
- Keys are **SHA-256 hashed** before storage — plaintext never persisted
- Key shown once at registration; cannot be retrieved later

### WordPress Admin
- All admin AJAX endpoints verify WordPress nonce (`crawlguard_admin_nonce`)
- `current_user_can('manage_options')` check on every admin request
- Public challenge endpoint uses per-challenge nonce (`crawlguard_challenge_{id}`)

---

## 2. JS Challenge (Proof-of-Work)

### Design
- Proof-of-work puzzle: find nonce where `SHA-256(challenge_id + nonce)` starts with N zero bits
- Difficulty: N = 10–20 (configurable), default 16
- Prevents trivial bot bypass — requires actual computation

### Verification Flow
1. Bot detected → challenge page served with unique `challenge_id`
2. Browser computes proof-of-work in JavaScript loop
3. Answer submitted via AJAX with WordPress nonce
4. Server verifies:
   - Nonce valid (anti-CSRF)
   - Transient exists for `challenge_id` (anti-replay, 5-min TTL)
   - SHA-256 hash meets difficulty requirement
   - Transient deleted after verification (one-time use)
5. On success: IP-bound HMAC cookie set

### Cookie Security
- Name: `crawlguard_verified`
- Value: `{timestamp}:{HMAC-SHA256(timestamp:client_ip, secret_key)}`
- TTL: 1 hour
- Flags: `HttpOnly`, `Secure` (if SSL), `SameSite=Lax`
- IP-bound: cookie invalidated if client IP changes

---

## 3. Rate Limiting

### Plugin-Side (WordPress)
- Sliding window per IP per minute (default: 60)
- Sliding window per IP per hour (default: 1000)
- Sliding window per User-Agent per minute (default: 120)
- Stored in `wp_crawlguard_rate_limits` table
- Exceeded → `429 Too Many Requests` with `Retry-After` header

### Worker-Side (Cloudflare)
- Per API key: 1000 requests/minute
- Per IP (unauthenticated): 60 requests/minute
- Enforced at edge before any database calls

- Plugin checks for `Signature` header on incoming requests
- Extracts `keyId` from signature parameters
- Looks up public key from `wp_crawlguard_signing_keys` table
- Verifies signature using `openssl_verify` with SHA-256
- **Currently log-only**: verification failure is logged but does not block
- Designed for future enforcement mode

### Supported Algorithm
- `rsa-sha256` (RSA PKCS#1 v1.5 with SHA-256)

## 🔴 The Problem

Every day, AI companies like OpenAI, Anthropic, Google, and Meta deploy sophisticated web crawlers — **GPTBot**, **ClaudeBot**, **Google-Extended**, **Bytespider**, and dozens more — that systematically scrape website content to train large language models worth billions of dollars.

**The creators get nothing.**

- **$0 in compensation** for content that powers AI products generating billions in revenue
- **Increased server costs** from high-frequency bot traffic that degrades site performance
- **No visibility** into which AI bots visit, how often, or what they take
- **No control** beyond a basic `robots.txt` that most aggressive crawlers ignore
- **No legal enforcement** — the current web ecosystem has no payment infrastructure for AI crawling

Content creators are subsidizing the AI industry with their intellectual property, their server resources, and their bandwidth — and they don't even know it's happening.

## 1. Executive Summary

**PayPerCrawl** is a SaaS platform that enables website owners to **detect, manage, and monetize AI bot traffic** crawling their content. It consists of three tightly integrated components:

| Component | Technology | Role |
|-----------|-----------|------|
| **WordPress Plugin** (CrawlGuard WP) | PHP 8+, WordPress 6.x | Client-side bot detection, JS challenge, admin UI, content scraping, WooCommerce sync |
| **Edge Backend** | Cloudflare Workers, JavaScript | API gateway, monetization engine, analytics aggregation, Stripe webhook handling |
| **Cloud Database** | Neon PostgreSQL | Persistent storage for sites, bot requests, payments, AI companies, analytics |

**Value Proposition:** Website owners install a single WordPress plugin, connect via API key, and immediately begin detecting AI bots (GPTBot, ClaudeBot, Google-Extended, etc.). Detected bots can be blocked, challenged with JS proof-of-work, or monetized via Stripe-powered paywalls — generating passive revenue from AI companies that scrape content.

## Key Development Workflows

### Adding a New Bot Signature

1. Edit `includes/class-bot-detector.php`
2. Add entry to `$known_bots` array: `'BotName' => ['company' => 'Company', 'type' => 'ai_crawler']`
3. If bot has special pricing, update `get_bot_multiplier()` in `backend/production-worker.js`

### Modifying Bot Detection Heuristics

Edit `includes/class-bot-detector.php` → `analyze_heuristically()`. Scoring:
- Each signal adds points (0–40 range)
- Threshold: ≥70 = classified as bot
- Signals: missing headers, suspicious UA patterns, datacenter IPs, scraping service headers

### Modifying Monetization Pricing

1. **Base price**: `sites.pricing_per_request` in PostgreSQL (default $0.001)
2. **Bot multipliers**: `analyzeBotRequest()` in `backend/production-worker.js`
3. **Content length bonus**: >5000 chars = +50%, >2000 chars = +25%
4. **Revenue split**: Stripe 2.9%+$0.30 / Platform 20% / Creator remainder
