## 数据来源 A：GitHub / GitLab PR 评论

### 方式一：导出 JSON（推荐）

GitHub API 导出单个 PR 的评论：

```bash
# 安装 GitHub CLI
brew install gh  # macOS
# 或 winget install --id GitHub.cli  # Windows

# 导出 PR 评论
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments > pr_comments.json

# 或使用 curl（替换 token 和 PR 信息）
curl -H "Authorization: token YOUR_TOKEN" \
  "https://api.github.com/repos/{owner}/{repo}/pulls/{pr_number}/comments" \
  > pr_comments.json
```

然后喂给 techboss-skill：
```
/create-techboss
# 选择 [C] 上传文件
# 提供 pr_comments.json 文件路径
```

### 方式二：截图直接上传

1. 打开 PR 页面，截包含 review 评论的截图
2. 在 `/create-techboss` 时选择 `[A] 上传截图`
3. AI 自动识别截图中的评基准字

### 方式三：复制粘贴文本

直接从 PR 页面复制评基准本，在 `/create-techboss` 时选择 `[B] 粘贴文本`。

**效果最大化技巧**：
- 优先提供：review 被 Request Changes 的评论（批评性最强，最能还原语言风格）
- 同时提供：他 approve 时的评论（呈现对比）
- 数量建议：至少 5-10 个 PR 的评论，覆盖不同项目

## 数据来源 D：会议纪要 / 录音转文字

### 架构评审会

最有价值的会议类型。包含他：
- 对技术方案的质疑方式（提什么问题、用什么大厂案例）
- 做决定的逻辑（真实技术考量 vs 个人偏好）
- 画饼时机（会议结尾的承诺）

如果有录音，可以用工具转文字：

```bash
# 使用 faster-whisper 转换（需要提前安装）

with open('meeting_transcript.txt', 'w') as f:
    for s in segments: f.write(s.text + '\n')
"
```

### 1v1 面谈记录

如果有会议记录（自己整理的），包含他的原话，直接粘贴。
重点关注他如何回应：
- 你提加薪时他怎么说
- 你提晋升时他怎么说
- 他批评你的技术时具体用了什么词

### Standup 会议

记录他在站会上问的问题，特别是：
- 提出与会议主题无关的技术话题
- 对谁的进度提出质疑
- 临时新增的"小需求"

## 常见问题

**Q: `/create-techboss` 没有反应？**

## 8 大流派鉴定

### 🔬 性能偏执派 PUA

**核心武器**：用复杂度和 benchmark 当道德武器，但从不提业务实际量级

**标志性话术**：
- "这个 O(n²) 在大数据量下会炸的"（当前数据量：50条）
- "你 profile 过吗？实际 hotpath 在哪知道吗？"
- "Redis 缓存呢？这个接口调用频率很高的"（调用频率：每天20次）
- "内存分配有没有考虑？这里会 GC 压力很大"

**PUA 精髓**：用正确的技术事实服务于错误的场景，让你永远在优化永远用不到的性能。

**受害者画像**：你开始对所有代码做 benchmark，线上服务 P99 早已达标但你还在优化。

---

### 🏛️ SOLID 传教士 PUA

**核心武器**：把软件工程原则变成宗教信条，违反一条就等于不会编程

**标志性话术**：
- "Single Responsibility 原则你了解吗？"（言下之意：你不懂）
- "Open/Closed Principle，这里改的方式不对"
- "这违反了 Dependency Inversion"
- "你的 interface 设计有问题，ISP 没有遵守"

**PUA 精髓**：把灵活运用原则的能力，变成对原则的字面服从，让工程师患上设计焦虑症。

**受害者画像**：你开始为每个函数写 Interface，为每个类拆 Abstract，代码量翻倍、业务速度减半。

---

### ☁️ 云原生布道者 PUA

**核心武器**：用"云原生"把一切简单问题复杂化

**标志性话术**：
- "为什么不用 Kubernetes？这个用 Docker Compose 不够"
- "Service Mesh 你们接入了吗？这个 observability 不够"
- "这个没有 Circuit Breaker，线上稳定性怎么保证"
- "你们的 GitOps 流程还没建立起来"

**PUA 精髓**：把"所有人都应该用的" 理想架构，当成"你们现在就必须有"的指责工具。

---

### 📚 设计模式考官 PUA

**核心武器**：GoF 23种设计模式随机连招，解决方案比问题更复杂

**标志性话术**：
- "这里用 Factory Method 比 switch-case 好多了"
- "Observer Pattern 你知道吗？不要轮询"
- "Template Method Pattern，抽象出来复用"

- "Airbnb 的 Data Infrastructure 文章你看了吗？"

**PUA 精髓**：用「我见过更好的」建立权威感，永远不解释那些做法为什么适合你们当前阶段。

---

### 🧹 代码洁癖患者 PUA

**核心武器**：用"clean"/"elegant"/"readable"等无法客观定义的词当review标准

**标志性话术**：
- "这段代码不够 clean，重写"（没说哪里 dirty）
- "命名不直观，改一下"（没说改成什么）
- "这不够优雅，感觉很 hacky"
- "Martin Fowler 的 Clean Code 你看了吗？"

**PUA 精髓**：把主观的审美偏好，包装成客观的技术标准，让你无法辩驳。

# Tech Counterattack — 技术反击话术库

## 前置分析

收到用户描述的 Tech PUA/攻击场景时，先分析：
1. **攻击类型**：代码评审 / 架构否定 / 甩锅甩到你 / 公开场合贬低 / 绩效打压
2. **攻击场合**：群聊 / PR 评论 / 评审会议 / 1v1 / 全员面前
3. **攻击用语**：提取关键词（scalability / tech debt / production ready / 不够 clean / 复杂度高等）
4. **用户当前状态**：忍 / 找后路 / 已决定走

---

## 三档反击框架

### 🟢 职场安全版 — "把问题结构化，球踢回去"

**核心逻辑**：
- 不争对错，把模糊批评变成可量化的问题
- 把行动项锚定在他的决定上
- 让他说出明确标准，而不是你猜

**万能句式**：
- "好的，我来整理一下 {技术指标} 的当前状态，我们对齐一下优化优先级。"
- "你说的 scalability 问题，能帮我指出具体场景吗？我来评估影响面。"
- "这个 tradeoff 当时是 {他批准的决策}，现在需要调整优先级吗？"
- "我来做一个 spike，评估重构成本，下次评审带方案来。"

**心理效果**：你显得专业、主动，他必须说出具体标准，同时给了你 2 周缓冲。

---

### 🟡 架构阴阳版 — "用他自己的话打他"

**核心逻辑**：
- 引用他之前批准的 ADR、Slack 消息、评审结论
- 用他提倡的原则来质疑他当前的要求
- 让在场其他人看到矛盾，但自己不直接宣战

**场景针对话术**：

**当他说"考虑过 scalability 吗"**：
> "我记得 {N 个月} 前的架构评审里，我们的结论是当前量级下 scalability 不是瓶颈，
> 优先 delivery。这个判断有变化了吗？我来更新一下 ADR。"

**当他说"这个 tech debt 不可接受"**：
> "这个 tech debt 是 {Sprint N} 里为了 {里程碑交付} 做的 tradeoff，
> 当时的结论记在 Jira #{ticket-id} 里。现在要还这个债，需要排进下个 Sprint 吗？"

**当他说"和现有架构不一致"**：
> "我对照了现有的 Architecture Decision Records，目前系统在 {组件 A} 上用的就是这个模式。
> 是新功能需要用不同的标准，还是需要更新一下 ADR？"

**当他说"测试覆盖率不够"**：
> "我们当前的 coverage 要求文档是多少？我按那个标准来补，避免以后反复对齐。"

---

### 🔴 摊牌版 — "证据链说话，技术决策有历史"

**核心逻辑**：
- 直接指出甩锅行为
- 用时间戳、会议记录、Jira 评论还原决策历史
- 点名技术决策的真实责任归属
- 仅限已决定走 / 已遭不公正绩效打压时使用

**框架**：
```
事实层："{这个问题}是{时间}在{场合}做的决策，决策者是{谁}。"
证据层："这里有{Slack截图/ADR文档/Jira评论}，可以复盘一下。"
责任层："如果这是现在的问题，需要我们一起在{RCA文档/复盘会}里对齐责任。"
行动层："我愿意参与复盘，但需要{具体条件：公开讨论/书面结论/更新绩效标准}。"
```

**摊牌示例**：
> "这个架构问题是 Q2 技术评审通过的方案，评审记录我还有。
> 如果现在产生了 production 问题，我们需要做一次正式 RCA，
> 把决策链条和责任归属写清楚。我可以发起这次 RCA。"

---

### 场景 A：PR 下大段否定评论

# Tech Predict — 技术老板心理预测器

## 预测框架

用于在技术评审 / 架构会议 / 1v1 / Sprint Review 前，预判 Tech Boss 的心理状态、反应模式和雷区。

---

## 预测维度

### 维度 1：当前心理状态评估

根据以下信号判断 Tech Boss 今天的状态：

| 信号 | 状态判断 |
|------|---------|
| 昨晚凌晨发了 Slack 消息 | ⚠️ 焦虑 + 微操模式启动 |
| 生产环境昨天出了 incident | 🔴 找替罪羊模式 + 重要决策风险高 |
| 面试了 3 个新人本周 | 🟡 对老人满意度降低，以新人为参照物 |
| 上午开完高层汇报 | 🟠 被上级压了，会把压力向下传导 |
| 刚看完某大厂技术博客 / 参加完技术会议 | 🟣 正在寻找机会"推广新技术" |
| 周一早上 | ⚡ 精力充沛，审判欲望高，彻底推翻一切的概率↑ |
| 周五下午 | 😴 想快点下班，只要过得去就能过 |
| 最近招聘难 | 🙂 需要留人，说话会软化一些 |

---

### 维度 2：会议类型 → 惯用战术

**技术评审会**：
- 必问：scalability、test strategy、rollback plan
- 爱甩：和上游系统的集成风险
- 画饼：会说"这个做完了我们可以开源"
- 地雷：提到"微服务"时容易失控开讲 30 分钟

**Sprint 规划**：
- 必压：排期（"这个一天够了吧"）
- 爱加：临时需求（"再加一个小功能"）
- 口头禅："MVP 思维，先跑起来"

**1v1 面谈**：
- 开场：永远先说公司大方向（拖时间）
- 甩锅预热：说团队挑战，暗示对你不满
- 大饼区间：承诺在最后 5 分钟出现
- 摊牌信号：说"我们聊聊你的 career path"

**线上 Incident 复盘**：
- 必找的替罪羊顺序：前端 → 后端 → 运维 → 数据库 → "基础设施问题"
- 必说的决定：CI/CD 流程要加强
- 不会提的：他上次审批跳过了 staging 环境测试

---

### 维度 3：议题 → 预期反应

**当你提出重构建议时**：
> 90% 概率："先把业务做出来，重构我们再安排"
> 10% 概率：被他劫持成他主导的"架构升级项目"

**当你提出要用新技术时**：
> 60% 概率：质疑稳定性和学习成本
> 30% 概率：如果他也想推这个技术，会突然变成支持
> 10% 概率：被他说"这个我早想引入了"然后变成他的 idea

**当你汇报进度慢于预期时**：
> 先说"这个有什么卡点？"（等你解释）
> 然后说"能不能删减功能先跑 MVP？"
> 然后说"这个我帮你协调资源"（下周没声音）

**当你提出加薪 / 晋升时**：
> 先说"公司今年形势不好"
> 再说"你还需要在 {技术方向} 有更多积累"
> 然后说"等这个项目上线我们认真谈"
> 项目上线后说"最近 HR Freeze 了，但我支持你的"

"""Parse email files (.eml/.mbox) to extract boss communications."""

import argparse
import email
import mailbox
import sys
from email.header import decode_header
from pathlib import Path


def decode_str(s):
    """Decode email header string."""
    if s is None:
        return ""
    decoded = decode_header(s)
    parts = []
    for content, charset in decoded:
        if isinstance(content, bytes):
            parts.append(content.decode(charset or "utf-8", errors="replace"))
        else:
            parts.append(content)
    return " ".join(parts)


def extract_body(msg):
    """Extract text body from email message."""
    if msg.is_multipart():
        for part in msg.walk():
            ct = part.get_content_type()
            if ct == "text/plain":
                payload = part.get_payload(decode=True)
                charset = part.get_content_charset() or "utf-8"
                return payload.decode(charset, errors="replace")
    else:
        payload = msg.get_payload(decode=True)
        if payload:
            charset = msg.get_content_charset() or "utf-8"
            return payload.decode(charset, errors="replace")
    return ""


def parse_eml(file_path: str, target: str) -> list:
    """Parse a single .eml file."""
    with open(file_path, "rb") as f:
        msg = email.message_from_binary_file(f)

    sender = decode_str(msg.get("From", ""))
    if target.lower() not in sender.lower():
        return []

    return [{
        "from": sender,
        "to": decode_str(msg.get("To", "")),
        "subject": decode_str(msg.get("Subject", "")),
        "date": decode_str(msg.get("Date", "")),
        "body": extract_body(msg),
    }]


def parse_mbox(file_path: str, target: str) -> list:
    """Parse an mbox file."""
    mbox = mailbox.mbox(file_path)
    results = []
    for msg in mbox:
        sender = decode_str(msg.get("From", ""))
        if target.lower() in sender.lower():
            results.append({
                "from": sender,
                "to": decode_str(msg.get("To", "")),
                "subject": decode_str(msg.get("Subject", "")),
                "date": decode_str(msg.get("Date", "")),
                "body": extract_body(msg),
            })
    return results


def main():
    parser = argparse.ArgumentParser(description="Parse boss emails")
    parser.add_argument("--file", required=True, help="Path to .eml or .mbox file")
    parser.add_argument("--target", required=True, help="Boss name to filter by")
    parser.add_argument("--output", required=True, help="Output file path")
    args = parser.parse_args()

    path = Path(args.file)
    if not path.exists():
        print(f"Error: file not found: {path}", file=sys.stderr)
        sys.exit(1)

    if path.suffix == ".mbox":
        emails = parse_mbox(str(path), args.target)
    else:
        emails = parse_eml(str(path), args.target)

    with open(args.output, "w", encoding="utf-8") as f:
        f.write(f"# Emails from {args.target}\n\n")
        f.write(f"Total: {len(emails)} emails\n\n")
        for i, e in enumerate(emails, 1):
            f.write(f"---\n## Email {i}\n")
            f.write(f"- From: {e['from']}\n")
            f.write(f"- To: {e['to']}\n")
            f.write(f"- Subject: {e['subject']}\n")
            f.write(f"- Date: {e['date']}\n\n")
            f.write(f"{e['body']}\n\n")

    print(f"Parsed {len(emails)} emails from {args.target} → {args.output}")


if __name__ == "__main__":
    main()

        sys.exit(1)

    for fname in ["SKILL.md"]:
        src = version_dir / fname
        if src.exists():
            shutil.copy2(src, boss_dir / fname)

    print(f"Rolled back {slug} to {version}")


def main():
    parser = argparse.ArgumentParser(description="Boss Skill version management")
    parser.add_argument("--action", required=True, choices=["backup", "rollback"])
    parser.add_argument("--slug", required=True)
    parser.add_argument("--version", default=None)
    parser.add_argument("--base-dir", default=".")
    args = parser.parse_args()

    if args.action == "backup":
        backup(args.slug, args.base_dir)
    elif args.action == "rollback":
        if not args.version:
            print("Error: --version required for rollback", file=sys.stderr)
            sys.exit(1)
        rollback(args.slug, args.version, args.base_dir)


if __name__ == "__main__":
